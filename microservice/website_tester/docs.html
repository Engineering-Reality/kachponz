Amadeus A2A Protocol v1

In-house Multi-Agent Coordination Protocol Documentation

How A2A Protocol Works

Amadeus A2A Protocol v1 is a state-of-the-art multi-agent coordination protocol designed internally for Bank Mandiri's Trade Finance settlement (Import LC/SKBDN/SBLC). It facilitates secure, air-gapped coordination between RPA robots (UiPath, PAD) and agentic LLMs.

Core Concepts

  • JSON-RPC 2.0: We utilize standard JSON-RPC request/response semantics over HTTP for clean error handling and structure.
  • Agent Card Discovery: A single entrypoint (`/.well-known/amadeus-agent-card.json`) broadcasts all available orchestrator capabilities, required parameters, and financial step flags.
  • Multi-turn Task Lifecycle: Every task has its own internal state machine (`submitted` → `working` → `input_required` → `completed` / `failed` / `canceled`). This allows for clarification loops without modifying the underlying transaction step in the database!
  • Live SSE Updates: Track task changes in real-time without polling via the `stream` endpoints.

End-to-End Task Flow

1. Robot calls `task.submit` to request task execution for a specific transaction step.
2. Orchestrator assigns a `taskId` and sets state to `submitted`.
3. An internal Executor picks it up → state transitions to `working`.
4. Executor encounters ambiguity → state transitions to `input_required`.
5. Robot (or human operator) provides input via `task.provideInput`.
6. Executor resumes processing → state back to `working`.
7. Task concludes successfully → state `completed` → Orchestrator triggers Transaction State Tracker to advance step.

Agents

Configuration-driven, secure intelligent actors for Trade Finance.

In Amadeus, an Agent is a secure, configuration-driven entity managed by a central LangGraph engine. Unlike traditional dynamic Python scripts, Amadeus Agents act as reasoning brains that determine which Model Context Protocol (MCP) Tools to execute without directly executing the code themselves.

Why use Agents in Amadeus?

Transitioning from rigid RPA workflows to dynamic, goal-oriented agentic workflows provides significant architectural advantages:

  • Dynamic Goal Orientation: Instead of strict step-by-step programming, Agents dynamically plan the path to achieve a goal (e.g., "Verify this LC document against the UCP 600 regulations").
  • Asynchronous Orchestration: Agents can seamlessly orchestrate legacy UiPath jobs, Power Automate Desktop (PAD) flows, and asynchronous OCR tasks without blocking the main event loop.
  • Resilience to Change: When UI elements change or APIs update, rigid RPAs break. Agentic workflows can dynamically adjust their plans, asking for clarification via the input_required state when they encounter ambiguity.

How it Works (Under the Hood)

Amadeus implements a Configuration-Driven approach designed strictly for air-gapped banking environments.

  • Stateful Graph Execution: Agents operate inside a stateless LangGraph engine. The entire execution graph is determined by the configuration, preventing arbitrary code execution.
  • Database-Backed Blueprints: Agent blueprints and system prompts are stored securely in PostgreSQL (a2a_tasks and transactions). They are not generated as dynamic code.
  • Model Context Protocol (MCP): Amadeus relies heavily on the MCP. The Agent reasoning engine never holds credentials. It simply outputs a structured JSON intent (e.g., "Extract text from Document X"). A standalone, highly-secured MCP Server then intercepts this intent, injects the necessary service accounts, and executes the physical tool.

Anatomy of an Agent

In the Amadeus Database, an Agent is represented not by a script, but by a structured JSON configuration block. This explicitly defines its reasoning constraints, allowed capabilities, and assigned LLM endpoint.

{
  "agent_id": "doc_exam_agent",
  "name": "Document Examination Agent",
  "system_prompt": "You are a Trade Finance analyst. Verify LC parameters against ICC UCP 600. Do not guess information. If a document is unreadable, fail the step with 'OCR_FAILURE'.",
  "allowed_mcp_tools": [
    "mcp.vision.extract",
    "mcp.core.get_transaction"
  ],
  "llm_config": {
    "provider": "dashscope",
    "model": "qwen-vl-max",
    "temperature": 0.1
  }
}

CISO Compliance Highlight: Secure Tooling Boundary

Because tools are executed over MCP, the LLM container never accesses the underlying DATABASE_URL or Core Banking API_KEY. If the LLM is compromised, it only has access to the isolated MCP tool intents explicitly whitelisted in its allowed_mcp_tools array.

Security & Compliance Gate

Security Note: High-Stakes Operations

Amadeus ensures that rogue or hallucinating agents cannot corrupt the ledger.

Agent execution is strictly protected by Optimistic Locking (version checking) and the A2A v1 Protocol. To alter the Transaction State Tracker (e.g., completing an LC step), the Agent's executor must satisfy a Dual Authentication requirement:

  1. Valid Bearer Token or X-Robot-Key.
  2. HMAC-SHA512 payload signature (Required for steps flagged with financial: true).

If an Agent attempts to jump a step or mutate a stale transaction, the Optimistic Lock will instantly reject the commit with a VERSION_CONFLICT or STEP_MISMATCH.

Testing via cURL

Before implementing the SDK inside UiPath or Python, you can test the entire A2A integration using simple cURL commands.

1. Discover Agent Capabilities

Fetch the Agent Card to understand what steps the orchestrator supports.

curl -s http://127.0.0.1:8080/.well-known/amadeus-agent-card.json | jq

2. Submit a Task

Requires an active `X-Robot-Key` and a valid `transactionId`.

curl -s -X POST http://127.0.0.1:8080/a2a/rpc \
  -H "X-Robot-Key: YOUR_ROBOT_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "test-req-1",
    "method": "task.submit",
    "params": {
      "transactionId": "6ff1204e-97c2-44dd-94da-4e3812b2152e",
      "step": "submitted",
      "correlationId": "test-curl-1"
    }
  }' | jq

Example Output:

{
  "jsonrpc": "2.0",
  "id": "test-req-1",
  "result": {
    "taskId": "c237be84-125a-4cbd-97a4-f1f6d9ac8078",
    "state": "submitted",
    "transactionId": "6ff1204e-97c2-44dd-94da-4e3812b2152e",
    "step": "submitted",
    "assigneeHint": null
  }
}

3. Fetch Task Status

Track the status of your task ID (including its message trail).

curl -s -X POST http://127.0.0.1:8080/a2a/rpc \
  -H "X-Robot-Key: YOUR_ROBOT_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "test-req-2",
    "method": "task.get",
    "params": {
      "taskId": "c237be84-125a-4cbd-97a4-f1f6d9ac8078"
    }
  }' | jq

4. Financial Signatures

For steps flagged as `financial: true` in the agent card (e.g., `mt_converted`), your cURL must include a secure HMAC-SHA512 payload signature to prevent Man-in-the-Middle attacks. Refer to the UIPath Developer guides for C# signature generation.
