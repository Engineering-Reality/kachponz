# Prompt #19 — Amadeus: Marketing Site Design System (Frontend) + Feature Wiring (Backend)

Two parts. Part A is the frontend design system — read this in full before writing
any CSS. Part B is what needs to be true on the backend/product side so the site's
screen recordings and feature demos show the REAL product, not mockups.

---

# Part A — Frontend Design System

## Do not touch these — they're already right

- The existing display/headline typeface and navbar typography. Do not swap fonts,
  do not change weights on the wordmark. This prompt is a **layout, motion, and
  decorative-accent** pass, not a typography redesign.
- The core brand colors already in use: indigo `#1E1B4B` (text/dark surfaces),
  fuchsia `#D946EF` (primary accent), lilac `#F3E8FF` (light surfaces). These stay
  as the foundation — everything below extends them, doesn't replace them.

## The one thing to avoid above all else

Read this list and actively steer away from every item on it — this is what makes a
site look AI-generated regardless of how much effort went into it:
- A warm cream background with a terracotta/clay accent (`#D97757`-ish) — a dead giveaway.
- A near-black background with exactly one flat neon accent color and nothing else.
- Every card using the same `rounded-xl shadow-md` treatment with no variation
  section to section — this is the single most common tell.
- Gradient text on headlines as a default move (purple-to-blue text fills) — way
  overused, reads as template, not brand.
- Centered-everything layouts with no asymmetry, no varied rhythm between sections.
- Numbered markers (01/02/03) used as decoration rather than because the content is
  a genuine sequence — only use these where order actually carries meaning (e.g. a
  real onboarding flow), not as a generic "features" layout device.

Reference the uploaded screenshots (Refold AI, Zite, LlamaIndex, LangSmith) for
**technique**, not for copying their specific compositions. What to actually borrow
from each, concretely:
- **Refold AI**: diagonal gradient color-bar accents used as a background texture
  behind hero copy, kept to the page's edges so they frame rather than compete with
  the headline. Bold black/dark headline typography with a single blue accent word.
- **Zite**: glassmorphic overlay panels (frosted, translucent, soft border) for
  interactive elements (search bars, input surfaces) floating over a photographic or
  gradient background.
- **LlamaIndex (LlamaParse)**: the numbered feature-showcase pattern (only if
  genuinely sequential — see above), product-screenshot cards with soft ambient
  shadow, a 3D isometric decorative object as a hero anchor.
- **LlamaIndex (homepage)**: the radiating gradient "soundwave" bar pattern as an
  ambient hero visual, orange→pink→violet→blue spectrum, symmetric and centered
  behind a focal 3D object.
- **LangSmith**: dark-mode glow — thin curved lines radiating/converging toward a
  focal point beneath the headline, client-logo trust strip, product screenshots
  with a soft blue glow edge in dark mode.

## Signature element — spend the boldness here, one bold move executed well

Call it the **Aurora Thread**: a single continuous animated gradient line/ribbon
(SVG path or canvas), using the spectrum below, that:
- Appears behind the hero as a soft, blurred, large-radius gradient mesh (this is
  the "RGB blur semburat warna" — the multi-hue blurred glow the brief asked for).
- Recurs, thinned down to a 1-2px glowing line, as a section-divider motif between
  major page sections — giving scroll-through continuity instead of hard stacked
  blocks (this is also the "DNA glow" feel — a single flowing thread, not random
  scattered particles).
- Gets a sparse particle/dot treatment ALONG the thread only (small glowing dots
  drifting slowly along the path) — not scattered freely across the whole page.
  This keeps "banyak rainbow partikel" contained to one coherent system instead of
  becoming visual noise.

**Aurora spectrum** (the "RGB Amadeus" palette — a curated cool-to-magenta spectrum,
not literal ROYGBIV, which would read as toy-like rather than premium):
```
--aurora-blue: #3B82F6
--aurora-cyan: #22D3EE
--aurora-violet: #8B5CF6
--aurora-fuchsia: #D946EF
```
Use as a 4-stop linear or conic gradient for the Aurora Thread and its blurred glow
variants. In dark mode, increase saturation and add a `filter: blur() + opacity`
glow layer behind the thread (matching LangSmith's dark-mode line-glow look). In
light mode, keep the thread more subtle — lower opacity, larger blur radius, more
like Refold's soft background texture than a bright neon line.

Implement the blurred hero variant as 2-3 large (600-900px) radial gradient blobs
using the aurora stops, `filter: blur(120px)`, positioned absolutely at low opacity
(15-25% in light mode, 25-35% in dark mode), with a very slow ambient drift
animation (60-90s loop, barely perceptible — atmosphere, not attention-grabbing).

## Layout plan — vary rhythm section to section, don't repeat one card grid

1. **Hero**: Aurora Thread blurred mesh background, existing headline typography
   (untouched), a glassmorphic CTA area (Zite-style), one live/animated preview
   element (see Part B) rather than a static screenshot.
2. **Trust strip**: grayscale logo row (LlamaIndex/LangSmith pattern) — if no real
   client logos exist yet, skip this section entirely rather than inventing fake
   logos.
3. **Feature showcase**: numbered ONLY if presenting the actual agent workflow
   sequence (Agent Creator → Tools Registry → Agent Invoke → Loop Mode is a genuine
   sequence, so numbering is justified here specifically) — screen-recording/video
   embeds per feature (see Part B for what these need to show), autoplay muted when
   scrolled into view, pause when scrolled out.
4. **Benchmark section**: the Amadeus-vs-Automation-Anywhere comparison — presented
   as an asymmetric two-column layout, not a plain table, to avoid the generic-grid
   tell. Consider a toggle/slider interaction rather than a static table.
5. **Architecture/how-it-works**: reuse the existing mermaid diagrams' visual
   language (the sequence flow, the 4-layer architecture) but re-rendered as
   interactive/animated SVG rather than a static embedded image — hover states on
   each layer revealing detail, matching the "interactive kalau di-scroll" ask.
6. **CTA close**: dark-mode Aurora Thread at full glow intensity as the page's final
   visual beat — bookending the hero's opening motif.

## Motion — scroll-driven, orchestrated, not scattered

- Section entrances: fade + slight upward translate on scroll-into-view
  (IntersectionObserver or a scroll-driven CSS animation), staggered for grouped
  elements (e.g. feature cards enter 80-120ms apart, not simultaneously).
- The Aurora Thread's line-divider segments should visually "draw on" (stroke-
  dashoffset animation) as the user scrolls past them, tying the animation to
  scroll position rather than just time — this is what makes it feel like the
  LangSmith/LlamaIndex references rather than a generic fade-in library default.
- Hover micro-interactions on interactive elements (buttons, feature cards) should
  intensify the aurora glow slightly rather than introducing a new effect —
  consistency with the signature element matters more than novelty per-component.
- Respect `prefers-reduced-motion` — disable the ambient drift and scroll-linked
  drawing animations, keep only essential state changes, for anyone with that
  preference set.

## Dark mode — a real second design, not an inverted palette

- Background: near-black indigo (`#0B0A1F` or similarly deep — deeper than the
  existing `#1E1B4B`, which should become a surface/card color IN dark mode, not
  the page background).
- The Aurora Thread and blob glows get MORE prominent in dark mode (this is where
  "darkmode-nya juga harus KEREN" gets delivered) — higher opacity, tighter blur
  radius on the line variant so it reads as a genuine glow source.
- Text: off-white with a faint lilac tint, not pure `#FFFFFF` (pure white on
  near-black is harsh — every reference site above uses an off-white).
- Product screenshot/video embeds get a soft colored border-glow (blue-violet, per
  the LangSmith reference) rather than a flat border.

## Responsive — nothing gets cut off, this is non-negotiable

- Test at 375px width minimum (not just 390/414 — 375 catches more overflow bugs).
- The Aurora Thread's blurred blobs must be clipped/repositioned per-breakpoint, not
  just scaled down — a 900px blur radius blob will overflow and get clipped weirdly
  on mobile if just left at desktop size; explicitly resize per breakpoint.
- Numbered feature showcase and the benchmark comparison section both need an
  explicit mobile layout (stacked, not a squeezed-down desktop grid) — design these
  as separate mobile compositions, not just `grid-cols-1` fallbacks of the desktop
  version.
- Video/screen-recording embeds: use `object-fit: cover` inside a fixed-aspect
  container so they never distort or overflow their card on narrow viewports.
- Test with real device emulation (not just browser resize) for at least one iOS
  and one Android viewport before considering this done.

## Build process (per the studio discipline this calls for)

1. Write the token plan first — color roles named, the Aurora Thread spec, layout
   concept per section as one-sentence descriptions — and self-review it against
   the "avoid above all else" list before writing any code.
2. Build section by section, screenshotting each at desktop AND 375px width as you
   go — don't build the whole page then discover mobile overflow at the end.
3. Final pass: check `prefers-reduced-motion`, keyboard focus visibility on every
   interactive element, and re-confirm nothing on the "avoid" list crept in.

# Part B — Feature Wiring (so the demos show the real product)

The marketing site's feature showcase (Part A, Section 3) needs actual screen
recordings of working features. Before recording anything, confirm each of these is
in a demo-able state:

1. **Agent Creator** — confirm the flow described in Prompt 12 (Loop Mode as an
   optional per-agent capability) is functional enough to record: create an agent,
   attach a tool, optionally configure a recipe step chain.
2. **Tools Registry / MCP integration** — record attaching an MCP tool to an agent
   and seeing it appear in that agent's available tools (ties to Prompt 13's
   namespacing work if multiple servers are shown).
3. **Agent Invoke / Loop Mode** — record an actual Danantara-CX100-style loop run
   (or a sanitized/demo equivalent if the real one touches production UiPath data)
   showing the structured progress view (iteration/step/variant), not just a raw
   chat transcript — this is the platform's clearest differentiator and deserves to
   be the centerpiece recording.
4. **Architecture visualization** — if Part A's interactive architecture diagram
   (Section 5) is meant to reflect live system state rather than a static diagram,
   that needs a lightweight read-only endpoint exposing current agent/tool
   counts or similar — confirm whether this is in scope before promising it in the
   design; a static (but interactive-on-hover) diagram is a perfectly reasonable
   fallback if a live-data endpoint isn't worth building for a marketing page.

For each recording: capture at a clean viewport size (1440px), in the SAME light/
dark theme as whichever mode the site defaults to, with realistic (not obviously
fake/placeholder) data — since one of this whole project's recurring lessons has
been that hallucinated or fake-looking state is exactly what erodes trust; the same
principle applies to marketing demos, not just the product itself.

## Acceptance criteria

- [ ] Existing typography (headline/navbar fonts) unchanged — diff confirms no font
      family/weight changes.
- [ ] Aurora Thread implemented as one coherent, reusable component (hero mesh, line
      divider, particle variants all derive from the same spectrum/system), not
      three unrelated gradient effects.
- [ ] Zero items from the "avoid above all else" list present anywhere on the page —
      explicit self-check before calling this done.
- [ ] Numbered markers appear only where content is a genuine sequence.
- [ ] Every section screenshotted at both desktop and 375px width with no overflow
      or clipping.
- [ ] `prefers-reduced-motion` disables ambient/scroll-linked animation.
- [ ] Dark mode background is a distinct near-black indigo, not an inverted light
      palette; Aurora Thread is visibly more prominent in dark mode.
- [ ] At least the Agent Invoke / Loop Mode recording shows real, working product
      behavior — not a static mockup image standing in for a "coming soon" feature.

## Non-goals

- Do NOT redesign the navbar or headline typography — explicitly out of scope per
  the brief.
- Do NOT invent placeholder client logos for the trust-strip section if no real
  ones exist yet — omit the section rather than fabricate social proof.
- Do NOT scatter the aurora/particle effect as a generic background texture on
  every section indiscriminately — it's a signature element with defined
  appearances (hero mesh, line divider, sparse thread-particles), not decorative
  wallpaper.
- Do NOT build the live-data architecture endpoint (Part B, item 4) unless Part A's
  design actually commits to live data over a static-but-interactive diagram —
  confirm which before building either.
